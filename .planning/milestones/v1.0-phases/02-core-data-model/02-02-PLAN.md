1. Create `lib/validations/availability.ts` with Zod schema:
   - `availabilityRuleSchema`: day_of_week (number 0-6), start_time (string HH:MM format), end_time (string HH:MM format), rule_type (enum: "available", "blocked", "preferred"), label (optional string, only for blocked type)
   - `availabilityRulesArraySchema`: array of availabilityRuleSchema with refinement that no two rules overlap on the same day

2. Create `app/onboarding/_components/AvailabilityGrid.tsx` ("use client"):
   - Visual 7-column (Sun-Sat) grid with time rows (6am to 11pm in 30-min increments — 34 rows)
   - Three interaction modes toggled by buttons at the top: "Available" (green), "Blocked" (red), "Preferred" (blue)
   - Click or click-drag on cells to paint them with the current mode. Click a painted cell to clear it.
   - When "Blocked" mode is active, show a small text input or dropdown to set a label (Work, Church, Class, Devotional, or custom text)
   - "BYU-Idaho typical" preset button: populates Mon-Fri 7-9am and 7-10pm as "available" slots. User can modify after applying.
   - State managed internally, exposed via `onChange(rules: AvailabilityRule[])` callback prop
   - Convert grid cells to availability_rules format: merge consecutive same-type cells in the same day into single start_time/end_time ranges (e.g., 3 consecutive "available" 30-min cells at 7:00, 7:30, 8:00 become one rule: 7:00-8:30)
   - Treat times as wall-clock times — no timezone conversion (per RESEARCH.md Pitfall 5)

3. Create `app/onboarding/_components/StepAvailability.tsx` ("use client"):
   - Renders `AvailabilityGrid` component
   - Manages rules state from grid onChange
   - "Save & Continue" button submits rules via `useActionState(saveAvailabilityRules, initialState)`
   - Rules passed as JSON in a hidden form input (the grid is not a native form — serialize the rules array)
   - Shows validation errors inline
   - On success, calls `onNext()` to advance to step 4

4. Add `saveAvailabilityRules` Server Action to `app/onboarding/actions.ts`:
   - "use server", authenticates with getUser()
   - Parses rules JSON from formData
   - Validates with availabilityRulesArraySchema
   - Uses delete-all + insert-new batch pattern: delete all existing availability_rules for this user, then insert the new set
   - Calls revalidatePath("/onboarding")
   - Returns { success: true }

5. Load existing availability rules in `app/onboarding/page.tsx` Server Component and pass to WizardShell/StepAvailability so the grid pre-fills on resume.
