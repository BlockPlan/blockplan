  <name>Task 1: Task validation schema and Server Actions (CRUD + status)</name>
  <files>
    lib/validations/task.ts
    app/tasks/actions.ts
  </files>
  <action>
1. Create `lib/validations/task.ts` with Zod schemas:
   - `taskSchema`: title (string, min 1, max 200), type (enum: "assignment", "exam", "reading", "other"), due_date (optional string, ISO date), estimated_minutes (optional coerced number, int, min 1, max 1440), course_id (string, UUID)
   - `taskUpdateSchema`: same fields as taskSchema but all optional (partial update), plus id (string, UUID, required)
   - Export both schemas, inferred types, and `DEFAULT_MINUTES` constant: `{ assignment: 120, exam: 180, reading: 60, other: 60 }`

2. Create `app/tasks/actions.ts` ("use server") with four Server Actions:

   a. `createTask(prevState, formData)`:
      - Authenticate with getUser()
      - Validate with taskSchema.safeParse(Object.fromEntries(formData))
      - Apply default estimated_minutes if not provided: use DEFAULT_MINUTES[type]
      - Verify course_id belongs to user (defense in depth with RLS) by querying courses table
      - Insert into tasks table with user_id, course_id, title, type, status="todo", estimated_minutes, due_date
      - revalidatePath("/tasks")
      - Return { success: true } or { errors: flatten().fieldErrors }

   b. `updateTask(prevState, formData)`:
      - Authenticate, extract task id from formData
      - Validate changed fields with taskUpdateSchema
      - Update in tasks table where id matches AND user_id matches (belt + suspenders with RLS)
      - revalidatePath("/tasks")
      - Return { success: true } or { errors }

   c. `deleteTask(taskId: string)`:
      - Authenticate with getUser()
      - Delete from tasks table where id = taskId (RLS ensures user_id match)
      - revalidatePath("/tasks")

   d. `updateTaskStatus(taskId: string, newStatus: "todo" | "doing" | "done")`:
      - Authenticate with getUser()
      - Update status field in tasks table
      - revalidatePath("/tasks")
      - This is the fast-path action for the inline status toggle — no Zod validation needed, just type assertion
  </action>

  <action>
1. Create `app/tasks/page.tsx` (Server Component):
   - Authenticate with getUser(), redirect to /auth if not authenticated
   - Check if user has any courses — if none, show message: "You need to add courses before creating tasks" with link to /onboarding or /courses
   - Read `searchParams` (MUST await — Next.js 16 async searchParams): course, type, status, sort
   - Build Supabase query: `supabase.from("tasks").select("*, courses(id, name)").eq("user_id", user.id)`
   - Apply filters from searchParams:
     - If `course` param: `.eq("course_id", course)`
     - If `type` param: `.eq("type", type)`
     - If `status` param: `.eq("status", status)`
   - Apply sort from searchParams:
     - `sort=due_date` (default): `.order("due_date", { ascending: true, nullsFirst: false })`
     - `sort=course`: `.order("courses(name)")`  — or sort client-side if Supabase doesn't support relation ordering
     - `sort=status`: `.order("status")`
     - `sort=created`: `.order("created_at", { ascending: false })`
   - Also fetch user's courses list (for the filter dropdown and task form course selector)
   - Pass tasks and courses to TaskList and TaskFilters components

2. Create `app/tasks/_components/TaskFilters.tsx` ("use client"):
   - Receives `courses` list and current filter/sort state from searchParams
   - Filter controls: Course dropdown (all courses + "All"), Type dropdown (assignment/exam/reading/other + "All"), Status dropdown (todo/doing/done + "All")
   - Sort control: dropdown with options "Due Date", "Course", "Status", "Recently Added"
   - On change, update URL searchParams using `useRouter().replace()` with new params — this triggers a Server Component re-render with fresh data
   - Use `useSearchParams()` to read current state and keep dropdowns in sync
   - Clear filters button that removes all searchParams
